# LingoLearn - Платформа для изучения языков
## Моя история создания этой платформы

Когда я начал работу над LingoLearn, моя цель была создать не просто еще одно приложение для изучения языков, а комплексную платформу, которая бы сочетала научно обоснованные методы обучения с геймификацией и гибкой архитектурой. Я хотел показать, как можно применять паттерны проектирования в реальном проекте, делая код не только функциональным, но и элегантным, расширяемым и поддерживаемым.

Вот как я реализовал каждый аспект этой системы, шаг за шагом:

## 🎯 Алгоритм SM-2: Моя математическая основа

Первым делом я внедрил алгоритм интервального повторения SM-2, потому что без надежной математической основы геймификация была бы бессмысленной. Я создал файл `core/utils/math_model.py` с двумя ключевыми функциями:

- `calculate_sm2()` - вычисляет новый фактор легкости и интервал на основе качества ответа
- `get_next_review_date()` - определяет дату следующего повторения

Самое интересное было интегрировать это в Django ORM. Я создал модель `SpacedRepetition`, которая хранит для каждого слова пользователя:
- Текущий фактор легкости (easiness factor)
- Количество правильных ответов подряд
- Дату следующего повторения

В `core/views.py` в методе `submit_answer()` я обновляю эти данные после каждого ответа пользователя, делая обучение truly адаптивным.

## 🏗️ Паттерны проектирования: Как я строил архитектуру

### Абстрактная фабрика для создания заданий

Одной из моих главных гордостей стала система генерации упражнений через абстрактную фабрику. Я хотел, чтобы добавление нового типа задания не требовало изменения существующего кода - принцип открытости/закрытости в действии.

В `core/patterns/abstract_factory.py` я создал:

1. **AbstractExerciseFactory** - абстрактный класс с методами для создания различных типов упражнений:
   - `create_multiple_choice()`
   - `create_translation()`
   - `create_matching_pairs()`
   - `create_listening()`

2. **VocabularyExerciseFactory** - конкретная фабрика, которая реализует создание упражнений для лексических пар

3. **Система декораторов** для модификаторов заданий - здесь я особенно горжусь своим решением:
   - `TimedExerciseDecorator` - добавляет ограничение по времени
   - `HintExerciseDecorator` - предоставляет подсказку
   - `ShuffleOptionsDecorator` - перемешивает варианты ответов
   - `AudioSupportDecorator` - добавляет аудио-поддержку для упражнений на аудирование

Прелесть этой системы в том, что я могу комбинировать декораторы:
```python
factory = VocabularyExerciseFactory("English", "Russian")
base_exercise = BaseExercise(factory.create_translation(entry))
modified_exercise = TimedExerciseDecorator(
    HintExerciseDecorator(
        ShuffleOptionsDecorator(base_exercise),
        "Подумайте о контексте"
    ),
    30  # 30 секунд
)
```

### Аудирование: Моя реализация listening exercise

Для упражнений на аудирование я создал специальную реализацию в фабрике:
- В `create_listening()` я перевожу исходное и целевое слово местами
- Добавляю поля `audio_text` и `transcript` для работы с аудио
- Использую `AudioSupportDecorator` для маркировки упражнения как имеющего аудио-поддержку

На фронтенде в `frontend/src/App.tsx` я проверяю наличие этого поля и подключаю Web Speech API для воспроизведения аудио.

### Наблюдатель и Система достижений

Когда я реализовывал систему достижений, я понял, что прямая проверка условий в коде приведет к сплетанию ответственности. Тогда я применил паттерн Наблюдатель.

В `core/patterns/behavioral.py`:

1. **Observer** - абстрактный класс с методом `update()`
2. **Subject** - класс, который хранит список наблюдателей и оповещает их
3. **AchievementObserver** - конкретный наблюдатель, который проверяет условия достижений
4. **AchievementDefinitionAdapter** - адаптер, который синхронизирует достижения с внешней системой

Как это работает в practice:
- После каждого submitted ответа я создаю `ProgressEvent`
- Этот событие передается в `Subject.notify()`
- Все наблюдатели (в текущей реализации - AchievementObserver) получают событие
- Observer проверяет все достижения пользователя против их условий
- Если условие выполнено и достижение еще не получено - оно разблокировывается
- За разблокировку достижения пользователь получает XP奖励

Особенно интересно получилось с адаптером. Я создал `LegacyExternalSystem` - симуляцию старой системы, из которой мы получаем определения достижений в неудобном формате. `AchievementDefinitionAdapter` преобразует эти данные в формат, понятный нашей системе, и синхронизирует их с базой данных через `update_or_create()`.

### Команда: Изолирование сложных операций

Для инкапсуляции процесса проверки ответа я использовал паттерн Команда.

`SubmitAnswerCommand` инкапсулирует:
- Ссылку на workflow (шаблонный метод прохождения упражнения)
- Данные ответа пользователя

Это позволяет мне:
- Откладывать выполнение операции
- Логировать команды для отладки
- Возвращаться к предыдущим состояниям (если понадобится)
- Тестировать процесс проверки ответа изолированно

### Другие паттерны, которые я применил

**Стратегия** для валидации ответов:
- Разные типы упражнений требуют разных способов проверки
- Я создал отдельные стратегии для множественного выбора, перевода и т.д.
- Это избавило меня от巨大的 условных операторов в коде проверки

**Состояние** для управления прогрессом курса:
- Курс может быть в состоянии активного, завершенного или приостановленного
- Каждое состояние имеет свое поведение при попытке перейти к следующему уроку

**Синглтон** для настроек приложения:
- Вместо постоянного чтения из файла или базы данных я создал синглтон `AppSettings`
- Это гарантирует, что настройки загружаются только один раз и доступны глобально

**Шаблонный метод** для стандартного рабочего процесса:
- Процесс прохождения упражнения имеет стандартные шаги: показать задание -> получить ответ -> проверить ответ -> обновить прогресс -> дать обратную связь
- Некоторые шаги могут быть переопределены для разных типов упражнений

## 🖥️ Фронтенд: Как я связал всё вместе

На фронтенде я применил те же принципы:

**Proxy** для кеширования API-запросов:
- В `frontend/src/patterns/patterns.ts` я создал `ApiProxy`
- Он перехватывает запросы к API и кеширует ответы
- Это значительно сократило количество запросов к бэкенду и улучшило отзывчивость интерфейса

**Composite** для структуры курсов:
- `CourseComposite` позволяет мне uniform работать с курсами, уроками и упражнениями
- Независимо от того, работаю ли я с целым курсом или отдельным упражнением, я использую тот же интерфейс

**Итератор** для перебора упражнений:
- `ExerciseIterator` предоставляет стандартный способ перебора упражнений в уроке
- Это избавило меня от написания однотипных циклов везде, где нужно пройтись по упражнениям

## 💾 База данных и начальное заполнение

Я спроектировал модели так, чтобы они отражали реальные сущности процесса обучения:
- `Course` - курс языка
- `Lesson` - урок внутри курса
- `Exercise` - конкретное задание
- `Enrollment` - запись пользователя на курс
- `ProgressLog` - журнал попыток пользователя
- `SpacedRepetition` - данные алгоритма SM-2 для каждого слова
- `Achievement`, `UserAchievement` - система достижений
- `UserProfile` - профиль пользователя с XP, серий и т.д.

Для быстрого старта я создал скрипт сидирования `core/management/commands/seed_db.py`, который генерирует:
- 8 языков интерфейса
- 15 уникальных курсов
- По 10 уроков в каждом курсе
- По 5-10 упражнений в каждом уроке
- Более 500 заданий с разными типами и уровнями сложности
- Предустановленные достижения для тестирования системы

**Этап 2: Паттерны порождающего и структурного типа**

## 📚 Что я получил в итоге

Платформа LingoLearn стала для меня демонстрацией того, как теоретические знания о паттернах проектирования и алгоритмах обучения могут превратиться в практический, работающий продукт. Каждый паттерн, который я применил, решил конкретную архитектурную проблему:

- **Абстрактная фабрика** сделала систему расширяемой для новых типов упражнений
- **Декораторы** позволили гибко модифицировать задания без создания экспоненциального количества подклассов
- **Наблюдатель** отделил логику достижений от основного процесса обучения
- **Адаптер** обеспечил интеграцию с hypotethical внешними системами
- **Командa** инкапсулировала сложные операции, сделав их тестируемыми и откатываемыми
- **Стратегия, Состояние, Синглтон** решили другие специфические задачи в коде

Самое gratifying чувство пришло, когда я увидел, как система работает в целом: пользователь отвечает на вопрос → алгоритм SM-2 обновляет интервал повторения → система проверяет достижения через наблюдателя → frontend обновляет график активности и серий → всё это происходит благодаря чистой, модульной архитектуре, которую я построил паттерн за паттерном.

Это проект показал мне, что хорошая архитектура не замедляет разработку - она ускоряет её на późniejszych этапах, делая код предсказуемым, легким в отладке и приятным в поддержке.
